# Configuration

Sythoria is designed to work out of the box with reasonable defaults, but it offers deep customization options to tailor its behavior to your specific development and privacy workflows. All settings are stored locally on your machine.

---

## Model Selection & Health Checks

You can switch your active LLM at any time using the **model dropdown** located directly in the chat input bar. The models shown in this dropdown are managed under **Settings > Model Providers**.

### Customizing Model Configurations

1. Open **Settings** and scroll to **Model Providers**.
2. Click **Add Model** to define a new endpoint.
3. Select a **Provider Preset** (OpenAI, Anthropic, Gemini, Ollama, etc.) to auto-populate the endpoint URL and model name, or choose **Custom** to enter your own OpenAI-compatible endpoint (e.g., LM Studio, Groq, Together AI, or a private vLLM server).
4. Fill in the **Model ID** matching the exact identifier expected by the provider (e.g., `gpt-4o` or `claude-3-5-sonnet-latest`).

### Connection Health Indicators

Each configured model displays a real-time health indicator dot:

- 🟢 **Green (Connected)**: The endpoint is active, reachable, and has successfully authenticated your API key.
- 🟡 **Pulsing Yellow (Connecting)**: A health check ping is currently executing.
- 🔴 **Red (Error)**: The health check failed. Hovering over the indicator will display the precise error code or message returned by the server.
- ⚪ **Gray (Inactive/Disabled)**: The model is disabled, or background connectivity checks are turned off.

### Background Health Check Logic

To ensure that models are ready when you need them, Sythoria performs automatic background health checks:

- **Interval**: Automatic checks are run every **5 minutes**.
- **Rate-Limiting (Throttling)**: Manual or automated check triggers are throttled to a minimum of **30 seconds** apart to prevent accidental API rate-limiting or provider abuse.
- **Power & Resource Savings**: If you want to prevent background network activity, toggle **Disable Background Activity** in Settings. This will halt all periodic health checks and force all indicators to gray until a model is manually selected or pinged.

## LLM Hyperparameters

You can fine-tune response characteristics globally or per model preset under the settings panel:

| Hyperparameter        | Range          | Default | Details                                                                                                                                                                              |
| :-------------------- | :------------- | :------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Temperature**       | `0.0` - `2.0`  | `0.7`   | Controls randomness. Lower values (`0.0` - `0.3`) make output deterministic, ideal for coding and facts. Higher values (`1.0` - `2.0`) encourage creative writing and brainstorming. |
| **Max Tokens**        | `1` - `4096+`  | Varies  | Restricts the maximum length of the model's generated response.                                                                                                                      |
| **Top P**             | `0.0` - `1.0`  | `1.0`   | Nucleus sampling. Filters candidate tokens based on cumulative probability mass. Setting to `0.9` means only tokens in the top 90% probability are considered.                       |
| **Frequency Penalty** | `-2.0` - `2.0` | `0.0`   | Discourages the model from repeating the same exact words or phrases.                                                                                                                |
| **Presence Penalty**  | `-2.0` - `2.0` | `0.0`   | Encourages the model to introduce and discuss new topics.                                                                                                                            |

## System Prompts & Repository Instructions (`AGENTS.md`)

System prompts dictate the personality, constraints, and operational guidelines of the AI model. Sythoria implements a layered system prompt compiler:

### 1. Global Default System Prompt

Configured under **Settings > General**, this prompt is applied to all new chats. You can use it to set general preferences, such as: _"Respond in concise markdown, avoid preambles, and use TypeScript for coding tasks."_

### 2. Workspace Context Injection (`AGENTS.md`)

If you mount a local folder as a developer workspace, Sythoria inspects the root of that folder for a file named `AGENTS.md`. If detected:

- The contents of `AGENTS.md` are **automatically read** and appended directly to the model's system prompt prior to every message sent in that workspace.
- This allows you to define project-specific conventions, code styling rules, or API schemas that the AI will follow seamlessly without manual copying.

## Developer Workspace Mounts (Project Tools)

Sythoria allows you to turn your chat interface into a local coding agent by mounting project folders. When a workspace is open, the model is granted access to a set of project tool primitives (e.g., directory listing, file searching, file reading, and editing).

### Workspace Permission Levels

To protect your file system, you must configure a security permission level when mounting a folder:

| Permission Level | Permitted Tools                                                                          | Security / HITL Behavior                                                                                                                                            |
| :--------------- | :--------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Read-Only**    | `project_glob`, `project_grep`, `project_list_dir`, `project_read`, `project_git_status` | **Auto-approve**: The model can navigate and inspect your workspace code instantly.                                                                                 |
| **Write/Commit** | _All Read-Only tools_ + `project_write`, `project_edit`, `project_git_commit`            | **Human-In-The-Loop**: The app pauses and shows a diff viewer when the model attempts to write or replace text. Files are only modified when you click **Approve**. |
| **Full (Bash)**  | _All Write tools_ + `project_bash`                                                       | **Strict Await Consent**: The app displays the command line string that the model wants to run in your terminal. You must review and click **Execute** to run it.   |

## Model Context Protocol (MCP) Configuration

Sythoria includes complete client support for the Model Context Protocol (MCP), enabling integration with local and remote developer toolkits.

### MCP Transport Protocols

Configure servers in **Settings > MCP Settings**:

- **Stdio**: Spawns local binaries or runtimes (e.g., `npx`, `uvx`, `python`, `docker`) using standard input/output streams.
  - _Parameters_: You must define the **Command** (e.g., `npx`) and **Arguments** (e.g., `-y`, `@modelcontextprotocol/server-postgres`) separately.
- **SSE (Server-Sent Events)**: Connects to an external running server over HTTP.
- **Streamable HTTP**: Lightweight HTTP-based remote server connections.

### MCP Security & Network Configurations

- **Opt-In Local Network Access**: By default, Sythoria blocks connections to private networks and loopback interfaces (e.g., `localhost`, `127.0.0.1`, RFC 1918 subnets) to prevent Server-Side Request Forgery (SSRF) attacks. If your MCP server runs locally, you must explicitly enable the **Allow Local Network** toggle in its configuration.
- **Trust Settings**:
  - **Untrusted (Default)**: Sythoria prompts you with a confirmation dialog showing arguments before running any tool from this server.
  - **Trusted**: The model is allowed to run tool calls from this server automatically, streamlining agent loops.
- **Environment Variables**: Custom environment variables needed by stdio servers are stored securely in your OS keychain alongside your LLM API keys.

## Web Search APIs

Sythoria can search the web and fetch page content to answer questions with real-time data. You can configure and toggle search engines under **Settings > Web Search**:

- **Google Custom Search**: Requires a Google API key and a Custom Search Engine ID (CX ID).
- **SearXNG**: Requires only the base URL of your self-hosted or public SearXNG instance.
- **Firecrawl**: Requires a Firecrawl API key, which enables optimized markdown extraction from search and scrape operations.
- **Custom Search APIs**: Custom endpoints utilizing OpenAI-compatible search schemas.

> [!TIP]
> You can toggle web search on or off dynamically during a conversation using the search icon directly next to the chat compose box.

## Application Constraints & Input Limits

To prevent application rendering lag and ensure memory stability, the following limits are enforced inside the desktop wrapper:

- **Max Message Length**: `10,000` characters per single chat submission.
- **Max File Upload Size**: `10 MB` per file.
- **Max Draft Attachments**: A maximum of `6` files (images, documents, code files) can be attached to one prompt.
- **Supported Text File Formats**: `.txt`, `.md`, `.json`, `.yaml`, `.yml`, `.csv`, `.log`, `.ini`, `.conf`.
- **Supported Image Formats**: `.png`, `.jpeg`, `.jpg`, `.gif`, `.webp`.

## Data Management & Storage Locations

Sythoria is completely stateless on the server side. All data is saved on your local hard drive.

### Filesystem Storage Locations

Settings, chat history databases, and cache folders are located in the user application support directory:

- **macOS**: `~/Library/Application Support/com.sythoria.sythoria-desktop/`
- **Windows**: `%APPDATA%\com.sythoria.sythoria-desktop\`
- **Linux**: `~/.local/share/com.sythoria.sythoria-desktop/`

### Configuration Files Structure

Within the storage directory, the following JSON files contain your data:

- `sythoria-store.json`: Contains your chat threads, conversation histories, UI preferences, keybinds, and MCP/search server connection endpoints.
- `config.json`: Contains model provider configurations (excluding credentials). Sensitive API keys are stripped out before saving to this file, as they are delegated directly to the OS keychain.
- `projects.json`: Stores details of your local project workspaces and their permission boundaries.

### Exporting and Resetting Data

- **Export Chat**: You can export individual conversations to markdown using the export icon in the active chat header.
- **Clear All Chats**: Located in the sidebar menu. Removes your database of chat histories, but keeps your configurations and API keys.
- **Factory Reset**: Located at the bottom of the Settings panel. This deletes all conversation histories, deletes configurations, and purges all credentials saved in the OS-level credential vault.
