# Privacy & Security

Privacy isn't a feature in Sythoria — it's the foundation. As a local-first, desktop-only interface, Sythoria is architected so that you have absolute control over your keys, configurations, and conversations.

Here is a technical deep-dive into how your data is handled, stored, and protected.

---

## Local Stateless Operation

Sythoria operates as a local-first application. Unlike conventional AI chat platforms that route your traffic through proprietary middleware servers, Sythoria establishes **direct peer-to-endpoint connections** from your device to the designated AI providers.

<DirectRequestFlow />

- **No Proxy Middlemen**: Your prompts, responses, files, and workspace context never pass through Sythoria-owned servers or database layers.
- **Stateless Request Flow**: Once a request is assembled and dispatched, the associated credentials exist in-memory only for the duration of the HTTP connection and are never logged or stored.
- **Direct API Streams**: Server-Sent Events (SSE) and WebSockets connect directly to provider endpoints, avoiding transport interception.

## Secure API Key & Credential Storage

Sensitive credentials—including provider API keys, search engine credentials, Model Context Protocol (MCP) authentication headers, and custom server environment variables—are never stored in plaintext configuration files.

Sythoria utilizes platform-native secure credential vaults to encrypt and isolate your keys.

### Platform-Native Integration

Under the hood, Sythoria's secure storage layer interfaces with the following native OS vaults:

- **macOS**: Apple Keychain Services (via the Security framework).
- **Windows**: Windows Credential Manager (using the Data Protection API / DPAPI).
- **Linux**: freedesktop.org Secret Service API over D-Bus (automatically interfacing with GNOME Keyring or KWallet).

### Secure Storage Implementations

Depending on your desktop platform runtime configuration, Sythoria implements the following cryptographic best practices:

#### 1. Electron `safeStorage` Architecture

For Electron-based deployments, the application integrates the built-in `safeStorage` API rather than deprecated third-party libraries (like `keytar` or `node-keytar`).

- **Cryptographic Encryption**: `safeStorage` uses Chromium's `OSCrypt` engine, utilizing **AES-128-CBC** encryption.
- **OS-Bound Key Derivation**: The decryption key is derived directly from the OS-level credential store. This protects against **at-rest data extraction**: if an adversary copies your configuration files from disk, they cannot decrypt the secrets without the platform's hardware/OS-bound key.
- **Linux Backend Validation**: On Linux systems, the app verifies the active backend (`safeStorage.getSelectedStorageBackend()`) to prevent falling back to insecure `basic_text` storage, warning the user if a valid D-Bus keyring is not detected.

#### 2. Tauri Keyring & Rust Security

For Tauri-based deployments, Sythoria interfaces with the OS keychain using the Rust `keyring` crate with `sync-secret-service` features.

- **Cryptographic Enclaves**: Utilizes secure hardware or cryptographic plugins like `tauri-plugin-stronghold` to build encrypted key-value databases on top of local storage.
- **Memory Zeroization**: The application implements the Rust `zeroize` crate to systematically wipe decrypted API keys from system RAM immediately after the request payload is dispatched, mitigating potential memory-scraping attacks.

## Runtime Sandboxing & Execution Isolation

Sythoria enforces strict process isolation to ensure that compromised frontend components or external dependencies cannot access your host OS or filesystem.

### Frontend-to-Backend Isolation

- **Context Isolation & Preload Scripts**: In Electron environments, the application runs with `contextIsolation: true` and `sandbox: true`. The renderer process (the frontend) has absolutely no direct access to Node.js APIs or system resources.
- **Tauri IPC Command Validation**: In Tauri, the WebView is completely isolated. It communicates with the Rust Core solely through a strongly-typed Inter-Process Communication (IPC) bridge. Every IPC command is bound to strict parameter validation schemas.
- **Content Security Policy (CSP)**: The app applies a strict CSP that blocks the execution of inline scripts and limits network requests exclusively to configured AI provider domains and local loopbacks.

### Tool & Subprocess Isolation

When utilizing developer integrations (such as local Git operations or MCP servers):

- **Directory Scoping**: Filesystem operations are strictly bound to folders you explicitly register as workspaces. The backend rejects attempts to read or write files outside these directories.
- **Isolated Subprocesses**: Local Model Context Protocol (MCP) servers run as isolated child processes. They are spawned with a minimized environment variable footprint, preventing secrets from leaking into the child process space.

## Zero Telemetry & Privacy Warranties

We believe that privacy policies are only as good as the code enforcing them. Sythoria implements **Zero Telemetry** by default:

- **No Remote Tracking**: There are no Google Analytics, Plausible, or tracking cookies embedded in the app.
- **No Crash Report Harvesting**: Unlike tools that use Sentry or LogRocket to upload full stack traces (which frequently contain sensitive prompts or API keys), Sythoria stores crash logs strictly on your local disk.
- **No AI Training**: Because your data never hits our servers, we can never use your conversations or code to train AI models. Your interactions remain entirely your own.

## File Storage Locations

All user-configured layouts, chat histories, and non-sensitive options are stored in plain JSON files. You can inspect, back up, or delete these files at any time.

### Storage Paths

- **macOS**: `~/Library/Application Support/com.sythoria.sythoria-desktop/`
- **Windows**: `%APPDATA%\com.sythoria.sythoria-desktop\` (resolving to `C:\Users\<Username>\AppData\Roaming\com.sythoria.sythoria-desktop\`)
- **Linux**: `~/.local/share/com.sythoria.sythoria-desktop/` (adhering to the `$XDG_DATA_HOME` specification)

### Database and Configuration Files

- `sythoria-store.json`: Contains local chat history, themes, and search/MCP configurations. **API keys and secrets are systematically stripped before writing to this file.**
- `config.json`: Stores configured model parameters and provider endpoints.
- `projects.json`: Registers local folder directories mapped to workspaces, along with their access permissions.

## Absolute Offline Privacy: Local Models

For users with strict air-gapped security requirements, Sythoria offers first-class compatibility with local model servers:

- **Ollama**: Connect to locally hosted models (e.g., Llama 3, Mistral) via `http://localhost:11434`.
- **Custom Local Endpoints**: Integrate LM Studio, Llama.cpp, or vLLM instances.
- **Air-Gapped Operation**: Once models are downloaded, you can disable all network connections. Sythoria will continue to function fully locally, ensuring zero bytes ever leave your machine.

For setup details, see the [Ollama Configuration Guide](/docs/providers/ollama).
